# v0.1 Agent Runner Design

Date: 2026-06-09

## Goal

Build a local runner for the counselor assistant v0.1 workflows.

The runner turns the current eval-only pipeline into a usable local agent path: a counselor can provide real input, choose a workflow, retrieve the relevant RAG chunks, call DeepSeek, save the prompt package and outputs, then run the same clean/rubric checks used by eval.

This is the product prototype runtime layer. It is not a web backend, frontend, Word export layer, or automatic clinical decision system.

## Context

The project already has:

- v0.1 workflows for intake form generation, case summary, and session note generation.
- RAG chunks and `rag/retrieval-map.v0.1.json`.
- DeepSeek API eval runner for fixed eval prompts.
- Output cleaner and rubric scorer.
- Eight workflow eval cases that currently pass clean and rubric checks.

The missing layer is a local command that accepts new user input instead of only running fixed eval prompt files.

## Scope

In scope:

- Run one v0.1 workflow against arbitrary local user input.
- Require explicit workflow selection in the first version.
- Load workflow instructions, relevant RAG chunks, and user input into a prompt package.
- Call DeepSeek using the existing local `.env` configuration pattern.
- Save a complete run folder under `agent-runs/`.
- Save input, prompt package, raw output, clean output, metadata, and safety/rubric result.
- Reuse existing clean/rubric logic where practical.
- Add tests with fake API transport; tests must not call the real DeepSeek API.

Out of scope:

- Automatic intent/workflow classification.
- Multi-turn chat memory.
- Frontend or backend HTTP API.
- Word or template export.
- Audio transcription.
- User-uploaded template parsing.
- Final clinical risk level judgment.
- Real-time crisis intervention automation.

## Supported Workflows

The first version supports three explicit workflow aliases:

```text
W1 / intake       初访信息收集表生成
W2 / case         个案信息整理
W3 / session      Session 总结与咨询记录
```

The runner should normalize aliases to internal workflow ids.

The first version does not infer workflow from the user's text. If the user gives the wrong workflow, the result should be treated as a user/operator error rather than an agent routing failure.

## Command Interface

PowerShell wrapper:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\run-agent.ps1 -Workflow W3 -Input "来访者本次谈到..."
```

Python entry point:

```powershell
python scripts\run_agent.py --workflow W3 --input "来访者本次谈到..."
```

Recommended options:

```text
--workflow W1|W2|W3|intake|case|session
--input TEXT                    Inline user input.
--input-file PATH               Read user input from UTF-8 text/markdown file.
--run-dir PATH                  Optional explicit output folder.
--dry-run                       Build and save prompt package without calling DeepSeek.
--no-clean                      Save raw output only; skip clean/rubric step.
--model NAME                    Override DEEPSEEK_MODEL for one run.
```

The command must require exactly one of `--input` or `--input-file`.

## Run Folder

Each successful command creates a timestamped folder:

```text
agent-runs/
  2026-06-09-143012-W3/
    input.json
    prompt_package.txt
    raw_output.txt
    clean_output.md
    metadata.json
    safety_check.json
```

`agent-runs/` should be ignored by git because it may contain real counseling material.

### input.json

Records sanitized run inputs and settings:

```json
{
  "workflow": "W3",
  "workflow_name": "Session 总结与咨询记录",
  "input_source": "inline",
  "user_input": "...",
  "created_at": "2026-06-09T14:30:12+08:00"
}
```

This file may contain sensitive user-provided content, so it stays local and untracked.

### prompt_package.txt

The exact prompt sent to DeepSeek.

It should contain:

- Agent role and global ethical boundaries.
- Workflow-specific task instructions.
- Retrieved RAG chunk excerpts.
- Required output structure for the selected workflow.
- User input.
- A completion marker such as `AGENT_DONE_W3`.

### raw_output.txt

The model response as returned by the API.

### clean_output.md

The cleaned final answer, suitable for human review.

### metadata.json

Records non-secret runtime information:

```json
{
  "status": "success",
  "model": "deepseek-v4-flash",
  "provider": "deepseek",
  "latency_ms": 1234,
  "token_usage": {
    "prompt_tokens": 0,
    "completion_tokens": 0,
    "total_tokens": 0
  }
}
```

The API key must never be written to metadata, logs, or prompts.

### safety_check.json

Records the local clean/rubric check result:

```json
{
  "status": "PASS",
  "rubric_status": "PASS",
  "issues": []
}
```

If a generic workflow-level rubric is not enough for free-form agent runs, the first version should still run basic checks:

- No deterministic diagnosis phrases such as `确诊为` or `诊断为`.
- Required workflow sections are present.
- Risk information is separated when relevant to the workflow.
- Missing information is marked as `未提供`, `未提及`, or `待补充`.
- Boundary language is present.

## Prompt Assembly

The runner should not reuse fixed eval prompt files, because eval prompts contain test case framing and eval markers.

Instead, create a reusable prompt builder:

```text
system/global instructions
  + workflow instruction block
  + RAG excerpts selected from retrieval-map
  + output contract
  + user input
```

Recommended source files:

- `counselor-agent-v0.1-system-prompt.md`
- `mvp-v0.1-workflows.md`
- `rag/retrieval-map.v0.1.json`
- referenced RAG chunk files under `rag/`

The first version can use a deterministic workflow-to-RAG mapping rather than semantic retrieval. Semantic retrieval can be added later.

## RAG Selection

The runner should select chunks by workflow:

- W1: intake assessment, forms/fields, informed consent, confidentiality, risk screening.
- W2: biopsychosocial assessment, case recording, risk boundary, deidentification when requested in the input.
- W3: session notes, risk change documentation, record keeping, confidentiality.

The selected chunk ids and file paths should be recorded in `metadata.json`.

If a referenced RAG file is missing, the runner should fail before calling the model and write an error metadata file.

## DeepSeek Configuration

Use the existing configuration pattern:

```text
DEEPSEEK_API_KEY=
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-flash
DEEPSEEK_TIMEOUT_SECONDS=120
```

Default model remains `deepseek-v4-flash`.

The runner can share config loading and HTTP request helpers with `scripts/run_model_eval.py` if that keeps the code simple. If sharing makes the eval runner harder to understand, use a small duplicate wrapper first and refactor later.

## Error Handling

Missing API key:

- Exit with a clear local setup message.
- Do not call the API.
- Do not print the key.

Invalid workflow:

- Show accepted workflow aliases.
- Do not create a partial run folder unless `--run-dir` was explicitly provided.

Missing input:

- Explain that exactly one of `--input` or `--input-file` is required.

RAG load failure:

- Write `metadata.json` with `status: "error"` and `error_type: "rag_load_error"` if a run folder exists.
- Do not call DeepSeek.

API failure:

- Write `metadata.json` with `status: "error"`, `error_type`, and sanitized message.
- Do not write fake clean output.

Cleaner/rubric warning:

- Keep `raw_output.txt` and `clean_output.md`.
- Write warning details to `safety_check.json`.
- Exit with a non-zero code only if the caller requested strict mode in a future version. The first version can return zero when the API call succeeds, because warnings are review signals.

## Testing

Tests should cover:

- Workflow alias normalization.
- Exactly-one input validation.
- Prompt package includes global instructions, workflow instructions, selected RAG excerpts, and user input.
- Dry-run creates input, prompt package, and metadata without calling API.
- Fake successful API response writes raw output, clean output, metadata, and safety check.
- Missing RAG chunk fails before API call.
- Missing API key fails without leaking secrets.
- Agent run outputs do not write into `eval-results/`.

No automated test should call the real DeepSeek API.

## Acceptance Criteria

The design is implemented when:

- `scripts/run_agent.py` exists.
- `scripts/run-agent.ps1` exists.
- `agent-runs/` is ignored by git.
- A dry run can build a prompt package for W1, W2, and W3.
- A real W3 call can produce a full run folder with raw output, clean output, metadata, and safety check.
- The runner records selected RAG chunk ids/paths in metadata.
- Existing eval runner behavior remains unchanged.
- Existing tests pass.
- New runner tests pass without a real API key.

## Future Extensions

4. Add uploaded Word template parsing and slot filling.
