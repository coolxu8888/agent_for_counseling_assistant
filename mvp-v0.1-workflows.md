# 咨询师助理 Agent MVP v0.1 工作流

## 文档目的

本文档用于把 MVP v0.1 的三个 P0 能力拆解成可执行的 Agent 工作流。每个工作流描述：

- 触发意图
- 输入识别
- 缺省策略
- 是否需要追问
- RAG 调用位置
- 处理步骤
- 输出结构
- 边界与异常处理

MVP v0.1 的三个 P0 能力共用一个原则：先建立流派无关的基础信息抓取系统。CPS/中国本地伦理与法律资料在本阶段主要作为伦理、风险、记录规范和专业边界参考；境外专业资料只作为补充参考并标注地域限制。CBT、人本、精神动力、家庭系统等理论框架留到后续“个案概念化”和“咨询方案设计”能力中使用。

# Workflow 1：初访信息收集表生成

## 1. 工作流目标

当咨询师要求生成初访信息收集表时，Agent 应生成一份统一、专业、流派无关的初访信息收集工具。

该工具可以根据用户需求输出为：

1. 咨询师访谈版
2. 来访者预填写版
3. 系统字段版

如果用户未指定版本，默认输出“咨询师访谈版”。

## 2. 触发意图

当用户表达以下需求时，进入本工作流：

- 生成初访信息收集表
- 生成初访访谈提纲
- 生成来访者预填写表
- 生成咨询师初访用问题清单
- 生成系统可落库的初访字段
- 生成咨询前信息收集模板
- 基于已有笔记生成后续需要补充询问的初访信息表

## 3. 典型用户输入

- 给我一个初访信息收集表
- 帮我生成一个咨询师初访访谈表
- 给来访者填的初访表
- 我要一个系统字段版的初访表
- 生成一个专业通用的初访信息收集表
- 帮我做一个咨询前信息收集模板
- 这是我已有的初访笔记，帮我生成还需要补充收集的信息表
- 根据下面这些笔记，帮我列一个下次初访继续问的提纲

## 4. 输入识别

Agent 需要从用户输入中识别以下信息：

| 信息项 | 是否必需 | 识别方式 | 默认值 |
|---|---|---|---|
| 表单用途 | 否 | 咨询师访谈、来访者预填写、系统字段、前端表单等关键词 | 咨询师访谈版 |
| 来访者类型 | 否 | 成人、青少年、儿童、伴侣、家庭等关键词 | 成人个体咨询 |
| 咨询场景 | 否 | 个体咨询、学校咨询、机构咨询、线上咨询等关键词 | 通用个体咨询 |
| 输出格式 | 否 | Markdown、JSON Schema、表格、字段配置等关键词 | Markdown 表格 |
| 特殊要求 | 否 | 简洁、温和、详细、加入知情同意、加入风险筛查等 | 使用标准完整版本 |
| 已有笔记或资料 | 否 | 用户粘贴了初访笔记、来访者资料、咨询师记录、转写文本等 | 无 |
| 笔记使用方式 | 否 | 生成空表、基于笔记生成补充提纲、标注已覆盖 / 待补充信息等 | 如有笔记且请求初访表，默认生成补充型信息收集表 |

## 5. 缺省策略

当用户只说“给我一个初访信息收集表”时，Agent 不需要追问，直接按以下默认值生成：

- 表单用途：咨询师访谈版
- 来访者类型：成人个体咨询
- 咨询场景：通用个体咨询
- 输出格式：Markdown 表格
- 风格：专业、清晰、克制
- 内容范围：包含基本信息、来访原因、当前困扰、生活背景、既往心理与医疗信息、风险评估、资源优势、咨询目标、知情同意和咨询师初步记录

当用户粘贴已有笔记，并明确要求“生成初访表 / 继续访谈提纲 / 补充信息收集表”时，Agent 不需要追问，默认生成“补充型初访信息收集表”：

- 保留统一基础结构
- 标注“笔记中已覆盖的信息”
- 标注“仍需补充询问的信息”
- 根据缺口生成后续访谈问题
- 不把咨询师笔记改写成诊断或概念化结论

## 6. 追问规则

本工作流默认少追问、多生成。只有在以下情况下才追问：

| 情况 | Agent 行为 |
|---|---|
| 用户明确要求“按我的机构模板生成”，但未提供模板 | 请用户粘贴模板或说明模板结构 |
| 用户要求“系统字段版”，但未说明输出格式 | 可默认输出 JSON Schema；如要接前端，可询问字段配置格式 |
| 用户要求儿童、青少年、伴侣或家庭咨询，但未说明使用对象 | 追问表单是给咨询师访谈使用，还是给来访者 / 监护人预填写 |
| 用户要求“特别简短”或“非常详细”，但目标不清 | 可直接给一个合理版本，并说明可继续压缩或扩展 |
| 用户只粘贴笔记，没有明确任务 | 询问用户是要“整理个案信息”，还是要“基于笔记生成补充初访访谈表” |
| 用户粘贴笔记并要求生成表单，但不清楚要空表还是补充型表单 | 默认生成补充型表单，并说明如需空表可另行输出 |

不需要追问的情况：

- 用户只要一个通用初访表
- 用户只说“来访者填的版本”
- 用户只说“咨询师访谈版”
- 用户只说“系统字段版”
- 用户粘贴笔记并明确要求“基于这些笔记生成后续访谈问题 / 补充信息收集表”

## 7. RAG 调用位置

本能力可以调用 RAG，但 RAG 的作用是提供专业底线和模板参考，不改变统一信息抓取结构。

| 工作流阶段 | 可调用 RAG | 用途 |
|---|---|---|
| 表单结构生成前 | 初访评估规范、机构初访表模板、生物-心理-社会模型 | 校验表单栏目是否完整，并覆盖来访者身体状态、心理状态和社会处境 |
| 风险评估栏目生成时 | 精神卫生法风险边界、CPS 伦理、风险评估规范 | 确保风险问题覆盖自伤、自杀、他伤、虐待、现实检验、物质使用等 |
| 知情同意栏目生成时 | CPS 保密、保密例外、知情同意原则 | 确保包含保密原则、保密例外、记录保存、危机服务边界 |
| 系统字段版生成时 | 字段规范、隐私分类规范 | 标注 sensitive、required、risk_signal 等字段 |
| 基于已有笔记生成补充表时 | 初访评估规范、个案记录原则、风险识别原则 | 对照统一结构识别已覆盖内容、待补充内容和必要风险追问 |

不调用的 RAG：

- CBT 个案概念化框架
- 人本主义咨询框架
- 精神动力概念化框架
- 家庭系统理论框架

如果用户要求“按 CBT / 人本 / 精神动力版本生成初访表”，Agent 应说明：

> 初访信息收集建议先保持统一，以便完整抓取来访者基础信息。CBT、人本、精神动力等流派框架更适合在后续个案概念化或咨询方案设计中使用。

随后可以继续生成统一表单，并可附加“后续概念化可补充探索的问题”。

## 8. 处理步骤

### Step 1：识别用户意图

判断用户是否在请求初访信息收集工具。

如果是，进入本工作流。

如果用户粘贴了笔记，并要求“基于笔记生成初访表 / 补充访谈提纲”，仍进入本工作流，但输出补充型信息收集表。

如果用户实际请求的是“根据已有来访资料做整理”，应转入 Workflow 2：个案信息整理。

如果用户只粘贴笔记、没有任务说明，应先澄清：

> 你希望我把这些笔记整理成个案信息摘要，还是基于它生成后续初访需要补充询问的信息表？

如果用户实际请求的是“根据一次 session 记录生成咨询记录”，应转入 Workflow 3：Session 总结与咨询记录生成。

### Step 2：识别表单版本

根据用户表达选择输出版本：

| 用户表达 | 输出版本 |
|---|---|
| 初访表、初访访谈、咨询师用 | 咨询师访谈版 |
| 来访者填、预填写、咨询前填写 | 来访者预填写版 |
| 系统字段、落库、JSON、前端配置 | 系统字段版 |
| 完整 MVP、三个版本、全部版本 | 同时输出三个版本 |
| 基于这些笔记继续问、补充收集、还要问什么 | 补充型初访信息收集表 |

### Step 3：套用统一基础结构

无论用户属于哪个流派，基础信息收集结构保持统一。

咨询师访谈版默认栏目：

1. 基本信息
2. 来访原因与咨询期待
3. 当前困扰评估
4. 重要生活背景
5. 既往心理与医疗相关信息
6. 风险评估
7. 个人资源与优势
8. 咨询目标初步设定
9. 咨询设置与知情同意确认
10. 咨询师初步记录

其中“当前困扰评估”和“重要生活背景”可按生物-心理-社会模型补充组织，用于覆盖身体状态、心理状态和社会处境。

### Step 4：根据版本调整语言

| 版本 | 语言要求 |
|---|---|
| 咨询师访谈版 | 专业、直接、便于访谈提问和记录 |
| 来访者预填写版 | 温和、非评判、允许跳过不适问题 |
| 系统字段版 | 字段清晰、命名稳定、标注敏感字段与风险字段 |

### Step 5：生成风险与知情同意栏目

风险评估必须覆盖：

- 自伤风险
- 自杀意念
- 自杀计划或准备
- 既往自伤 / 自杀尝试
- 他伤风险
- 暴力、虐待或不安全环境
- 物质使用风险
- 现实检验相关问题
- 保护因素
- 当前安全等级或待进一步评估

知情同意必须覆盖：

- 保密原则
- 保密例外
- 咨询频率、时长、费用、取消规则
- 咨询不是即时危机服务
- 记录保存与隐私保护方式
- 来访者是否同意开始咨询

### Step 6：输出最终表单

输出时优先给出可直接使用的表单。除非用户要求解释，否则不要在表单前写太长说明。

建议输出顺序：

1. 简短说明
2. 表单正文
3. 可选备注：该表单用于专业辅助，不替代咨询师判断

如果用户提供了已有笔记，建议输出顺序：

1. 简短说明
2. 笔记中已覆盖的信息
3. 待补充信息
4. 后续访谈问题表
5. 风险与知情同意补充检查项
6. 可选备注：基于笔记生成，仅供咨询师进一步访谈参考

## 9. 输出结构

### 9.1 咨询师访谈版


## 10. 边界处理

| 情况 | Agent 应如何处理 |
|---|---|
| 用户要求诊断量表 | 可以生成信息收集表，但应说明该表不是诊断工具 |
| 用户要求按 CBT / 人本等流派生成初访表 | 说明 v0.1 初访信息抓取保持统一，流派框架留到后续概念化 |
| 用户要求删掉风险问题 | 可以生成精简版，但应建议保留基本风险筛查 |
| 用户要求来访者必须填写所有问题 | 建议改为“可选择填写”，避免压迫性和伦理风险 |
| 用户要求收集过多隐私信息 | 建议遵循最小必要原则，只收集与咨询相关的信息 |
| 用户要求系统字段版但包含敏感信息 | 必须标注 sensitive: true |
| 用户要求系统自动判断高风险 | v0.1 只标注 risk_signal，不做自动风险分级或处置决策 |
| 用户直接粘贴笔记但未说明任务 | 不直接猜测输出，应询问是整理个案信息还是生成补充初访表 |
| 用户粘贴笔记并要求补充访谈 | 不做概念化解释，只识别已覆盖信息、待补充信息和后续访谈问题 |
| 笔记中包含高度敏感信息 | 只保留与咨询相关的最小必要内容，避免在输出中扩散无关隐私 |

## 11. 验收标准

该工作流可视为完成，当 Agent 能稳定做到：

1. 用户只输入“给我一个初访信息收集表”时，默认生成咨询师访谈版。
2. 用户要求来访者预填写版时，语言明显更温和，且允许跳过不适问题。
3. 用户要求系统字段版时，能输出稳定 JSON Schema，并标注 sensitive 和 risk_signal。
4. 风险评估栏目不缺失自伤、自杀、他伤、虐待、现实检验和物质使用相关问题。
5. 知情同意栏目包含保密原则、保密例外和危机服务边界。
6. 不因咨询师流派不同而改变基础信息抓取结构。
7. 信息不足时使用默认策略，而不是频繁追问。
8. 不将初访信息收集表写成诊断工具或治疗方案。
9. 用户提供已有笔记时，能生成补充型信息收集表，并区分已覆盖信息与待补充信息。
10. 用户只粘贴笔记且未说明任务时，能正确澄清而不是误入错误工作流。

# Workflow 2：个案信息整理

## 1. 工作流目标

当咨询师提供个案背景、初访记录、零散笔记或来访者资料时，Agent 应将这些材料整理成结构化个案信息摘要。

本工作流的目标是做“信息抓取与分类”，不是做流派概念化。Agent 需要区分事实、主诉、咨询师观察、初步假设、推测和待进一步评估内容，为后续个案概念化、方案设计和报告生成准备干净的原始材料。

## 2. 触发意图

当用户表达以下需求时，进入本工作流：

- 整理个案背景信息
- 整理初访笔记
- 提取主要问题、关系线索、风险信号
- 把零散记录整理成结构化摘要
- 根据来访资料形成个案信息摘要
- 用户直接粘贴一段个案资料，并要求“整理一下”

## 3. 典型用户输入

- 帮我整理这个来访者的背景信息
- 把这段笔录整理成结构化摘要
- 提取主要问题、关系线索和风险信号
- 帮我把这些零散记录整理一下
- 这是初访记录，帮我整理成个案摘要
- 下面是来访者资料，帮我看看目前有哪些信息和缺口

## 4. 输入识别

Agent 需要从用户输入中识别以下信息：

| 信息项 | 是否必需 | 识别方式 | 默认值 |
|---|---|---|---|
| 个案原始材料 | 是 | 用户粘贴的背景、笔记、转写文本、访谈记录、来访者资料等 | 无 |
| 整理目标 | 否 | 摘要、初访整理、风险线索、关系线索、信息缺口等关键词 | 结构化个案摘要 |
| 咨询阶段 | 否 | 初访、进行中、阶段总结、结案前等关键词 | 未知阶段 |
| 咨询流派 | 否 | CBT、人本、精神动力、家庭系统等关键词 | 仅记录为背景，不改变整理结构 |
| 输出格式 | 否 | 表格、条目、报告段落、SOAP 等关键词 | Markdown 条目 + 表格 |

## 5. 缺省策略

当用户提供个案材料并说“整理一下”时，Agent 不需要追问，默认输出“结构化个案信息摘要”。

默认输出包括：

- 主诉
- 当前困扰
- 关键事件
- 情绪与认知线索
- 行为模式
- 人际关系线索
- 资源与保护因素
- 风险信号
- 已知事实
- 初步假设
- 推测与待验证内容
- 信息缺口
- 建议进一步询问的问题

如果用户直接丢笔记但没有任何任务说明，Agent 应先做轻量澄清，而不是直接输出：

> 你希望我把这些笔记整理成个案信息摘要，还是基于它生成后续初访需要补充询问的信息表？

如果笔记明显是单次 session 记录，并且用户要求“生成咨询记录 / 总结本次 session”，应转入 Workflow 3。

## 6. 追问规则

本工作流默认少追问、多整理。只有在以下情况下才追问：

| 情况 | Agent 行为 |
|---|---|
| 用户只粘贴资料，没有说明任务 | 询问是要个案信息整理、补充初访表，还是 session 记录 |
| 用户要求按机构固定格式整理，但未提供格式 | 请用户粘贴模板或说明字段 |
| 用户要求按某一流派分析个案 | 说明本工作流只做基础整理；如要流派分析，应转入后续“个案概念化”能力 |
| 用户要求判断诊断或病名 | 说明不能做确定性诊断，可整理相关表现与待评估信息 |
| 材料过短，无法形成摘要 | 输出可提取内容，并列出待补充问题 |

不需要追问的情况：

- 用户明确要求“整理背景信息”
- 用户明确要求“提取主诉和风险”
- 用户明确要求“整理成结构化摘要”
- 用户明确要求“列出信息缺口”

## 7. RAG 调用位置

本能力可以调用 RAG 来校验整理维度、风险识别和记录边界。

| 工作流阶段 | 可调用 RAG | 用途 |
|---|---|---|
| 设定整理结构时 | 初访评估规范、个案记录规范、生物-心理-社会模型 | 确保摘要维度覆盖身体、心理和社会处境 |
| 提取风险信号时 | 精神卫生法风险边界、CPS 伦理、风险评估规范 | 标注自伤、自杀、他伤、虐待、现实检验、物质使用等风险线索 |
| 区分事实与推测时 | CPS 个案记录与专业实践原则 | 避免把推测写成事实 |
| 处理敏感信息时 | CPS 保密与记录保存原则 | 遵循最小必要原则，避免扩散无关隐私 |
| 生成外部分享 / 督导材料时 | CPS 去识别化原则、个人信息保护法敏感字段规则 | 泛化姓名、具体学校/单位、精确年级、地名、具体家庭结构和独特事件组合，降低再识别风险 |

不调用的 RAG：

- CBT 个案概念化框架
- 人本主义咨询框架
- 精神动力概念化框架
- 家庭系统理论框架

如果用户要求“按 CBT / 人本 / 精神动力来整理”，Agent 应说明：

> 我可以先把个案信息整理成统一基础摘要。CBT、人本、精神动力等流派理解更适合在后续“个案概念化”中使用。

## 8. 处理步骤

### Step 1：识别用户意图

判断用户是否在请求整理已有个案资料。

如果用户要求生成空白初访表，转入 Workflow 1。

如果用户要求基于已有笔记补充初访访谈问题，转入 Workflow 1 的“补充型初访信息收集表”。

如果用户要求总结单次 session 或生成咨询记录，转入 Workflow 3。

### Step 2：检查是否有可整理材料

如果用户没有提供材料，提示用户粘贴：

> 请把来访者背景、初访笔记或零散记录粘贴过来，我会按“事实、主诉、线索、风险、资源、信息缺口”的结构整理。

如果用户已提供材料，继续下一步。

### Step 3：抽取已知事实

先提取材料中明确出现的信息，例如：

- 人口学信息
- 来访原因
- 主要事件
- 时间线
- 既往咨询或医疗信息
- 当前功能受损情况
- 明确的风险描述

已知事实必须来自用户提供的材料，不得补写。

如果用户要求生成外部分享、督导群、教学案例或报告材料，应先做去识别转换，再进入后续整理：

- “小林 / 张某某” → “来访者 / 个案 A”
- “北师大 / 某具体院系 / 具体单位” → “某高校 / 某机构”
- “研二 / 大三 / 高三”等精确阶段 → “研究生阶段学生 / 高校学生 / 中学生”
- “父母离异”等具体家庭结构 → “重要家庭结构变化 / 家庭支持相关议题”
- 独特事件组合 → 保留功能性描述，避免完整复述可识别细节

去识别规则不仅适用于“已知事实”，也适用于“信息缺口”“建议进一步询问的问题”和“初步假设”。

### Step 4：提取临床相关线索

在不做诊断的前提下，整理：

- 情绪线索
- 认知线索
- 行为模式
- 人际关系线索
- 家庭与成长背景线索
- 压力源
- 资源与保护因素

### Step 5：标注风险信号

风险信号单独列出。即使材料中没有相关内容，也应标注：

> 材料中未见明确风险信息；建议咨询师按需进一步评估。

风险信号包括：

- 自伤
- 自杀意念、计划或准备
- 既往自伤 / 自杀尝试
- 他伤
- 暴力、虐待或不安全环境
- 物质使用
- 现实检验受损相关表现

### Step 6：区分初步假设与推测

初步假设必须使用谨慎语言，例如：

- 可能
- 似乎
- 初步看
- 需要进一步验证

推测不能写进“已知事实”。

### Step 7：列出信息缺口和后续问题

根据统一基础信息结构，列出还缺什么信息，并给出咨询师可进一步询问的问题。

5. 能单独列出风险信号。
6. 材料中没有风险信息时，能标注“未提供，建议按需评估”。
7. 能列出信息缺口和后续询问问题。
8. 用户只粘贴笔记但未说明任务时，能正确澄清。

# Workflow 3：Session 总结与咨询记录生成

## 1. 工作流目标

当咨询师提供某次 session 的记录、口述反馈、转写文本或笔记时，Agent 应生成结构化 session 总结、咨询记录和下次咨询重点。

本工作流的重点是“单次会谈记录沉淀”，不是完整个案概念化，也不是长期咨询方案设计。Agent 应帮助咨询师保存本次会谈的主题、关键内容、干预、来访者反应、风险变化、进展与阻滞，并形成可用于专业记录系统的文本。

## 2. 触发意图

当用户表达以下需求时，进入本工作流：

- 总结今天这次 session
- 根据记录生成咨询记录
- 提取本次咨询重点和下次方向
- 生成 SOAP / DAP / BIRP 记录
- 整理本次会谈中新出现的信息
- 根据本次咨询笔记生成后续重点

## 3. 典型用户输入

- 帮我总结今天这次 session
- 根据我的记录生成咨询记录
- 提取本次咨询重点和下次方向
- 帮我整理来访者这次的新信息
- 按 SOAP 格式生成咨询记录
- 下面是第 3 次咨询笔记，帮我生成记录和下次重点

## 4. 输入识别

Agent 需要从用户输入中识别以下信息：

| 信息项 | 是否必需 | 识别方式 | 默认值 |
|---|---|---|---|
| Session 原始记录 | 是 | 本次咨询笔记、转写文本、口述反馈、要点记录等 | 无 |
| 来访者基本背景 | 否 | 用户附带的基本资料或前情摘要 | 未提供 |
| 咨询阶段 | 否 | 初访、第几次咨询、阶段中、结案前等关键词 | 未知阶段 |
| 记录格式 | 否 | SOAP、DAP、BIRP、普通记录、机构模板等关键词 | 普通咨询记录 |
| 下次咨询需求 | 否 | 下次重点、访谈问题、家庭作业、风险跟进等关键词 | 输出下次咨询重点 |
| 新增信息 | 否 | 本次新出现的事件、风险、目标变化、关系变化等 | 从材料中抽取 |

## 5. 缺省策略

当用户提供 session 材料并说“帮我总结 / 生成咨询记录”时，Agent 不需要追问，默认输出“普通咨询记录 + 下次咨询重点”。

默认输出包括：

- 本次主题
- 来访者状态
- 关键内容
- 咨询师干预
- 来访者反应
- 风险变化
- 进展与阻滞
- 新增个案信息
- 已知事实
- 咨询师判断或初步假设
- 待补充信息
- 下次咨询重点
- 咨询记录正文

如果用户指定 SOAP、DAP 或 BIRP，则优先使用指定格式。

如果用户只粘贴笔记但没有说明任务，Agent 应先做轻量澄清：

> 你希望我把这些内容整理成个案信息摘要，还是生成本次 session 的咨询记录？

## 6. 追问规则

本工作流默认少追问、多生成。只有在以下情况下才追问：

| 情况 | Agent 行为 |
|---|---|
| 用户没有提供 session 材料 | 请用户粘贴本次 session 记录、转写文本或要点 |
| 用户只粘贴笔记，没有说明任务 | 询问是要个案信息整理，还是生成本次 session 记录 |
| 用户要求按机构模板生成，但未提供模板 | 请用户粘贴模板或说明字段 |
| 用户要求生成 SOAP / DAP / BIRP 以外的格式但未说明结构 | 先按普通咨询记录输出，并说明可按模板重排 |
| 材料中没有任何风险信息 | 不追问，输出中标注“未提供，建议咨询师按需评估” |

不需要追问的情况：

- 用户明确要求“总结本次 session”
- 用户明确要求“生成咨询记录”
- 用户明确指定 SOAP、DAP 或 BIRP
- 用户明确要求“提取下次重点”

## 7. RAG 调用位置

本能力可以调用 RAG 来校验记录格式、专业边界和风险记录要求。

| 工作流阶段 | 可调用 RAG | 用途 |
|---|---|---|
| 选择记录结构时 | 咨询记录模板、机构记录规范 | 确保输出可用于记录系统 |
| 生成 SOAP / DAP / BIRP 时 | SOAP / DAP / BIRP 格式说明 | 确保字段含义正确 |
| 记录风险变化时 | 精神卫生法风险边界、CPS 伦理、风险评估规范 | 确保风险变化单独列出 |
| 处理记录保存与隐私时 | CPS 保密与记录保存原则 | 避免扩散无关敏感信息 |

不调用的 RAG：

- CBT 个案概念化框架
- 人本主义咨询框架
- 精神动力概念化框架
- 家庭系统理论框架

如果用户要求“用 CBT / 人本 / 精神动力视角总结本次 session”，Agent 应说明：

> 我可以先生成统一的 session 记录。流派视角下的理解和反思更适合进入后续“个案概念化”或“方案设计”能力。

## 8. 处理步骤

### Step 1：识别用户意图

判断用户是否在请求单次会谈总结或咨询记录。

如果用户请求生成空白初访表，转入 Workflow 1。

如果用户请求整理来访者长期背景或个案资料，转入 Workflow 2。

如果用户请求按某一流派做解释、概念化或方案，提示该能力属于后续 v0.2。

### Step 2：检查是否有 session 材料

如果没有材料，提示用户粘贴：

> 请把本次 session 的笔记、转写文本或要点发来，我会整理成本次主题、关键内容、干预、反应、风险变化和下次重点。

如果已有材料，继续下一步。

### Step 3：抽取本次会谈事实

先整理材料中明确出现的信息：

- 本次主题
- 来访者自述
- 重要事件
- 咨询师观察
- 咨询师干预
- 来访者反应
- 新增背景信息

### Step 4：整理进展与阻滞

根据材料抽取：

- 本次进展
- 咨询中的卡点
- 来访者态度或参与度变化
- 咨询目标是否有变化

不补写材料中没有的信息。

### Step 5：单独记录风险变化

风险变化必须单独列出。

如果材料中有风险内容，按以下结构输出：

- 风险类型
- 材料中的具体信息
- 与上次相比的变化
- 建议咨询师进一步评估或处理

如果材料中没有风险内容，输出：

> 材料中未提供风险相关信息，建议咨询师按需进一步评估。

### Step 6：生成下次咨询重点

下次重点应基于本次材料，而不是凭空设计治疗方案。

可以包括：

- 下次需要继续探索的问题
- 需要跟进的风险或功能变化
- 需要澄清的信息缺口
- 可观察的变化指标

### Step 7：按指定格式生成记录

如果用户指定普通记录，输出普通咨询记录。

如果用户指定 SOAP，输出：

1. S：Subjective，来访者主观陈述
2. O：Objective，咨询师观察到的信息
3. A：Assessment，咨询师评估与初步理解
4. P：Plan，后续计划

如果用户指定 DAP，输出：

1. D：Data，本次资料与观察
2. A：Assessment，评估与理解
3. P：Plan，后续计划

如果用户指定 BIRP，输出：

1. B：Behavior，来访者表现与议题
2. I：Intervention，咨询师干预
3. R：Response，来访者反应
4. P：Plan，后续计划

该工作流可视为完成，当 Agent 能稳定做到：

1. 用户提供 session 材料后，能生成结构化 session 总结。
2. 能输出普通咨询记录、SOAP、DAP 或 BIRP 格式。
3. 能区分来访者自述、咨询师观察、咨询师判断和后续计划。
4. 能单独列出风险变化。
5. 材料中没有风险信息时，能标注“未提供，建议按需评估”。
6. 不添加咨询师未提供的信息。
7. 不把 session 总结写成诊断或完整治疗方案。
8. 用户只粘贴笔记但未说明任务时，能正确澄清。
